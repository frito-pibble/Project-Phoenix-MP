# Exercise Catalog & ID-First Resolution — Mobile

**Date:** 2026-05-03
**Status:** Approved
**Resolves:** [GitHub Issue #404](https://github.com/9thLevelSoftware/Project-Phoenix-MP/issues/404)
**Scope:** Project-Phoenix-MP (Kotlin/KMP — domain models, sync DTOs, SQLDelight, ExerciseImporter)
**Companion Spec:** `phoenix-portal/docs/superpowers/specs/2026-05-03-exercise-catalog-portal.md`
**Dependency:** Portal-side changes (Supabase migration, Edge Functions) must be deployed BEFORE this mobile update ships.

## Problem

Exercises with identical names but different equipment variants (e.g., "Bicep Curl" with Handles vs Long Bar vs Short Bar) lose their identity during the sync round-trip between mobile and portal. The portal stores exercises by name string only. When the mobile pulls data back, the three-tier name lookup (`findExerciseByName LIMIT 1`) returns an arbitrary variant, corrupting routines.

### Mobile-Side Root Causes

1. Sync DTOs send `exerciseId` for sessions but the portal ignores it
2. Routine sync DTOs don't consistently carry `exerciseId`
3. Sync pull resolution uses name-based lookup as the PRIMARY strategy — `findExerciseByName` with `LIMIT 1` returns whichever row the DB returns first for duplicate names
4. No `displayName` field — exercises with identical names are only distinguishable by their equipment subtitle in the UI
5. Some exercise names use trailing whitespace as a disambiguation hack (e.g., `"Alternating Lunges "`)

### What the Mobile Already Has Right

- `Exercise.id TEXT PRIMARY KEY` — stable IDs from `exercise_dump.json`
- `RoutineExercise.exerciseId` FK — exists but not carried through sync
- `ExerciseRepository.getExerciseById()` — direct ID lookup exists but isn't used on sync pull
- 1,176-exercise library with rich metadata (equipment, muscles, videos, etc.)

## Solution

Ensure exercise IDs survive the sync round-trip by carrying them in all DTOs, and switch sync pull resolution from name-first to ID-first. Add `displayName` field for human-readable disambiguation.

### Display Name Generation Rule

```
If only one exercise has this base name -> display_name = name
If multiple exercises share the base name -> display_name = "{name} ({primary equipment})"

"Primary equipment" = first item in the equipment array. The exercise_dump.json
consistently lists the most distinctive attachment first (e.g., ["LONG_BAR", "BENCH"]
-> "Long Bar" is primary). If the equipment array is empty, fall back to display_name = name.
```

## 3. ExerciseImporter Changes

### Display Name Generation

After parsing all exercises from `exercise_dump.json`, generate `displayName`:

```kotlin
fun generateDisplayNames(exercises: List<ParsedExercise>): Map<String, String> {
    val grouped = exercises.groupBy { it.name.lowercase().trim() }
    return exercises.associate { exercise ->
        val siblings = grouped[exercise.name.lowercase().trim()] ?: listOf(exercise)
        val displayName = if (siblings.size > 1) {
            val primaryEquipment = exercise.equipment.firstOrNull()
                ?.toDisplayLabel() ?: ""
            if (primaryEquipment.isNotEmpty()) {
                "${exercise.name.trim()} ($primaryEquipment)"
            } else {
                exercise.name.trim()
            }
        } else {
            exercise.name.trim()
        }
        exercise.id to displayName
    }
}

fun String.toDisplayLabel(): String = when (this.uppercase()) {
    "HANDLES" -> "Handles"
    "BAR" -> "Bar"
    "LONG_BAR" -> "Long Bar"
    "SHORT_BAR" -> "Short Bar"
    "ROPE" -> "Rope"
    "BELT" -> "Belt"
    "BENCH" -> "Bench"
    "STRAPS" -> "Straps"
    "GREY_CABLES" -> "Cables"
    else -> this.lowercase().replaceFirstChar { it.uppercase() }
}
```

### Trailing-Space Cleanup

During import, all exercise names are trimmed:

```kotlin
val cleanName = rawName.trim()  // "Alternating Lunges " -> "Alternating Lunges"
```

The `displayName` field handles disambiguation instead of whitespace hacks.

### Import Flow Update

```
1. Parse exercise_dump.json
2. Trim all names
3. Generate displayName for each exercise
4. Upsert into Exercise table with displayName column populated
```

## 5. Sync Pull — ID-First Resolution

### SqlDelightSyncRepository Changes

Replace the current name-first resolution with ID-first:

```kotlin
/**
 * Resolve an exercise from a sync pull DTO.
 * PRIMARY: Direct ID lookup (fast, unambiguous).
 * FALLBACK: Legacy three-tier name lookup (for old data without IDs).
 */
fun resolveExercise(
    exerciseId: String?,
    exerciseName: String,
    exerciseMuscleGroup: String
): Exercise? {
    // PRIMARY: Direct ID lookup — O(1), unambiguous
    exerciseId?.let { id ->
        exerciseRepository.getExerciseById(id)?.let { return it }
    }

    // FALLBACK: Legacy three-tier name lookup
    // Only reached for data synced before exercise_id was added
    return findExerciseByNameFallback(exerciseName, exerciseMuscleGroup)
}

/**
 * Legacy name-based resolution. Kept for backward compatibility with
 * data that was synced before exercise_id was added to the pipeline.
 */
private fun findExerciseByNameFallback(name: String, muscleGroup: String): Exercise? {
    // Tier 1: Exact name + muscle group
    database.findExerciseByNameAndMuscle(name.trim(), muscleGroup.trim())
        .executeAsOneOrNull()?.let { return it.toExercise() }

    // Tier 2: Exact name only
    database.findExerciseByName(name.trim())
        .executeAsOneOrNull()?.let { return it.toExercise() }

    // Tier 3: Case-insensitive
    database.findExerciseByNameCaseInsensitive(name.trim().lowercase())
        .executeAsOneOrNull()?.let { return it.toExercise() }

    return null
}
```

### Routine Pull Processing

When processing pulled routines:

```kotlin
fun processRoutinePull(dto: RoutineSyncDto) {
    dto.exercises.forEach { exerciseDto ->
        val exercise = resolveExercise(
            exerciseId = exerciseDto.exerciseId,       // NEW: use ID first
            exerciseName = exerciseDto.exerciseName,
            exerciseMuscleGroup = exerciseDto.exerciseMuscleGroup
        )
        // Upsert routine exercise with resolved exercise reference
        if (exercise != null) {
            database.upsertRoutineExercise(
                id = exerciseDto.id,
                routineId = exerciseDto.routineId,
                exerciseId = exercise.id,              // Resolved canonical ID
                exerciseName = exercise.displayName ?: exercise.name,
                exerciseMuscleGroup = exercise.muscleGroup,
                exerciseEquipment = exercise.equipment,
                // ... rest
            )
        }
    }
}
```

## 7. Rollout

### Dependency

The portal-side changes MUST be deployed first:
1. `exercise_catalog` table must exist in Supabase
2. Edge functions must accept and store `exerciseId` from DTOs
3. Edge functions must return `exerciseId` in pull responses

Only after those are live should the mobile update ship.

### Backward Compatibility

| Scenario | Behavior |
|----------|----------|
| New mobile + old edge functions | Mobile sends `exerciseId` in DTOs. Old edge functions ignore unknown fields. No regression, but IDs aren't stored. |
| New mobile + new edge functions | Full exercise_id flow. Bug #404 fixed. |
| Old mobile + new edge functions | Old mobile doesn't send `exerciseId`. Edge functions store NULL. Graceful degradation. |

### Success Criteria (Mobile Side)

1. Issue #404 resolved: routine with "Bicep Curl (Long Bar)" survives sync round-trip with correct equipment variant
2. All sync DTOs carry `exerciseId` from `Exercise.id`
3. Sync pull uses ID-first resolution — name lookup is fallback only
4. `displayName` populated for all exercises during import
5. Trailing-space name hacks cleaned up
6. No regression for users who haven't synced yet (name fallback still works)
